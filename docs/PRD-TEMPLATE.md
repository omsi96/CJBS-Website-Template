# PRD — <Feature / Project Name>

> A PRD answers **what** we're building and **why**, clearly enough that a developer (Claude) can
> build it and you can tell when it's done. Keep every requirement testable. Delete the guidance in
> quotes as you fill it in.

## 1. Summary
> One paragraph: what is this, for whom, and what changes for them?

## 2. Problem & motivation
> What pain or need exists today? Why is it worth solving now?

## 3. Persona(s)
> Who is this for? Link to the persona doc. Name their goal.

## 4. Goals & non-goals
- **Goals:** > the outcomes this must achieve
- **Non-goals:** > explicitly out of scope (prevents creep)

## 5. The core user journey
> Step by step, what does the user do, and what does the system do in response?
> 1. User …
> 2. System …

## 6. Requirements
> Numbered, testable statements. "The system shall …". Each must be verifiable.
- R1. …
- R2. …

## 7. Data
> What needs to be stored? Sketch the Prisma models (tables + key fields).

## 8. Success metrics
> How do we know it worked? (e.g. "a student can complete X in under N steps".)

## 9. Open questions
> Anything undecided. Resolve before building.
